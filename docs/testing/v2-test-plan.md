# 댕케어 캘린더 V2 TDD 테스트 설계서

본 문서는 TDD(Test-Driven Development) 관점에서 V2 신규 기능들의 정상(Happy Path) 및 예외(Edge Case) 동작을 검증하기 위한 테스트 케이스(Given-When-Then)입니다.

## 1. 체중 및 건강 트래커 (Health Tracker)

### [정상] 체중 기록 및 차트 렌더링
* **Given**: `healthRecords`에 과거 체중 데이터(5.2kg, 5.1kg)가 존재할 때
* **When**: 사용자가 오늘 날짜로 '5.3kg'을 신규 등록하면
* **Then**: `healthRecords` 배열에 새 항목이 추가되고, 꺾은선 차트의 가장 우측에 5.3kg 지점이 즉시 렌더링되어야 한다.

### [예외] 비정상적인 체중 값 입력
* **Given**: 체중 입력 폼이 활성화된 상태에서
* **When**: 사용자가 체중 값으로 '0', '음수(-2)', 혹은 '문자열(abc)'을 입력 후 저장하려 하면
* **Then**: 저장이 차단되고 "올바른 체중(양수)을 입력해주세요"라는 유효성 검사 에러 메시지가 노출되어야 한다.

## 2. 사진 및 메모 다이어리 (Diary)

### [정상] 사진이 포함된 다이어리 저장
* **Given**: 특정 캘린더 날짜를 선택한 상태에서
* **When**: 사용자가 3MB 크기의 사진을 첨부하고 메모를 작성하여 저장하면
* **Then**: 브라우저 단에서 이미지가 리사이징(압축)되어 Base64로 변환된 후 `diaryRecords`에 저장되며, 화면에 썸네일과 메모가 즉시 렌더링되어야 한다.

### [예외] Local Storage 용량 초과 방어
* **Given**: Local Storage 용량이 거의 가득 찬(4.9MB) 상태에서
* **When**: 사용자가 새로운 사진(압축 후 200KB)을 다이어리에 추가하려 하면
* **Then**: QuotaExceededError를 감지하여 저장을 취소하고, "저장 공간이 부족합니다. 이전 데이터나 사진을 삭제해주세요."라는 경고 모달을 띄워야 한다. 앱이 크래시되지 않아야 한다.

## 3. 주간/월간 통계 차트 (Statistics)

### [정상] 주간 루틴 달성률 계산
* **Given**: 이번 주 `records`에 전체 목표 10회 중 7회가 완료된 상태일 때
* **When**: 사용자가 '통계 탭'에 진입하면
* **Then**: 달성률 공식 `(완료된 횟수 / 총 목표 횟수) * 100`에 따라 70%가 계산되어 차트와 텍스트로 정확히 표출되어야 한다.

### [예외] 완료된 루틴이 전혀 없는 경우
* **Given**: 앱을 처음 사용하거나 루틴 체크를 한 번도 하지 않아 `records`가 비어있을 때
* **When**: 통계 탭에 진입하면
* **Then**: 차트 영역이 에러(NaN, Infinity 등)로 깨지지 않고, 0% 상태와 함께 "아직 달성한 루틴이 없어요. 오늘 루틴을 시작해볼까요?"라는 Empty State 화면이 안전하게 렌더링되어야 한다.
